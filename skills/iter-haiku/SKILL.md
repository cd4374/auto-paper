# iter-haiku

通用三段式迭代工作流。将任务拆分为 **Pre → Implement → Post** 三个节拍，Post 验收不通过时带着反馈回到 Pre 重新来过，直至满足用户原始需求或达到迭代上限。

## 参数

| 参数 | 说明 | 默认 |
|------|------|------|
| task | 用户原始需求描述 | 必填 |
| max_iterations | 最大整体迭代轮数 | 5 |

## 工作流

### 初始化

1. 固化 `original_task = {task}`，作为后续每一轮 Post 验收的**唯一基准**
2. 初始化 `ITER_LOG.md`，记录每轮迭代的关键决策与问题
3. 初始化迭代计数器 `iteration = 0`

### 迭代循环（三段式）

当 `iteration < max_iterations` 时，执行完整的三段式迭代：

---

#### 节拍一：Pre（需求解读与规划）

**目标**：基于当前已知信息生成本轮执行计划。

- **第 1 轮**：围绕 `original_task` 进行需求澄清、范围界定、约束识别，输出 `PLAN.md`
- **第 N 轮（N > 1）**：先读取上一轮 Post 阶段记录的未通过原因与问题列表，重新解读需求（问题是否源于理解偏差？ plan 是否需调整策略？），更新 `PLAN.md`

`PLAN.md` 必须包含：
- 本轮需求理解摘要
- 执行步骤清单（带里程碑）
- 验收标准（必须与 `original_task` 对齐）
- 本轮假设与风险

---

#### 节拍二：Implement（实现）

**目标**：严格按当前 `PLAN.md` 执行，不做计划外发散。

1. 逐条执行步骤清单
2. 每个里程碑完成后记录实际产出
3. 若执行中发现计划不可行，记录阻塞点，继续完成其余可行部分（不中途返回 Pre）

---

#### 节拍三：Post（验收审查）

**目标**：对照 `original_task` 审查当前实现是否满足用户原始需求。

执行验收审查，按以下格式输出审查结果：

**输入信息**：
- 用户原始需求：`original_task`
- 本轮计划：`PLAN.md` 内容
- 实际实现：当前完成的内容与产出

**审查步骤**：
1. 逐条对照 `original_task`，检查实现是否满足
2. 检查有无遗漏、偏差或过度实现
3. 评估质量是否达到可交付标准

**输出格式**：
```
## 验收结论
通过 / 不通过

## 满足项（如通过）
1. [需求点] → [实现对应]
2. ...

## 未通过原因（如不通过）
1. [问题描述及与原始需求的对照]
2. ...

## 回到 Pre 的建议（如不通过）
1. [需求重新理解或计划调整方向]
2. ...
```

**解析验收结果**：
- **通过** → 退出整体迭代循环，进入【完成阶段】
- **不通过** → 将未通过原因与建议写入 `ITER_LOG.md`，`iteration += 1`，回到【节拍一：Pre】

**循环终止条件**：
- Post 给出"通过"结论
- 或达到 `max_iterations` 轮上限（最后一次 Post 未通过也终止）

---

### 完成阶段

1. **输出最终交付物**：汇总实现结果
2. **输出 `ITER_LOG.md`**：包含每轮迭代的简要记录
   - 第 1 轮：Pre 理解 → Implement 产出 → Post 结论（通过/不通过 + 关键原因）
   - 第 2 轮：...
3. **标注状态**：
   - 若因"通过"而结束：提示任务完成
   - 若因达到 `max_iterations` 而结束：提示未解决问题，列出遗留问题，建议人工介入或扩大迭代上限

## 输出文件

| 文件 | 说明 |
|------|------|
| `PLAN.md` | 当前轮次的执行计划（每轮覆盖或追加版本号） |
| `ITER_LOG.md` | 迭代日志，记录每轮三节拍的结果与问题 |

## 关键实现要点

1. **原始需求不可变**：`original_task` 固化后不再修改，Post 审查必须始终对照它
2. **整体循环**：Post 不通过时回到 Pre，而非仅在 Post 内部修改。这是与 re-iter 的核心区别
3. **Pre 的演进性**：第 N 轮 Pre 必须显式读取上一轮 Post 的反馈，体现"带着问题重新理解需求"
4. **Prevent 死循环**：每轮迭代必须严格递增 `iteration`，达到上限即停止
5. **执行不中断**：Implement 阶段发现计划问题时不立即返回 Pre，先记录阻塞点并尽可能完成其余部分，把问题留给 Post 统一判断
