# 00-00-idea-brainstorm

- REVIEWER_MODEL = `gpt-5.5` — Model used via Codex MCP.
- MAX_POST_REVIEW_ROUNDS = 10 — Post-review 迭代轮数上限。

围绕用户给定的研究方向、约束与偏好，生成 `00-00-idea-pool.md`。

## 输入

- 用户描述的研究方向/主题
- 可选约束：算力/时间预算、偏计算/理论/数值模拟（或偏实验）、目标 venue、已有数据/代码/实验资产

## 输出

`00-00-idea-pool.md` -- 主要语言用中文，专业术语首次出现时用中文并括号标注英文原文（如"Transformer 架构（Transformer architecture）"）。

## 写作原则

00 阶段是**方向漏斗的入口**——输出必须精炼，只给方向性判断，不做展开。具体要求：

- **每个 idea 控制在 1-2 段（150-300 字）**：只写核心问题、关键洞见、为什么可能成立、最大不确定性。不展开方法论细节，不列完整风险清单，不写详细实验设计。
- **方向优先于细节**：读者需要知道的是"往哪走、为什么这个方向有意思"，而不是"具体怎么做"。方法路径只需要一句话的方向，实验只需要"能用 X 类数值模拟验证"这个粒度。
- **推理链紧凑**：每句话都应承载推理功能——要么在提出判断，要么在给出该判断的依据。去掉过渡句、铺垫、重复。
- **判断明确**：不用"具有一定的研究价值"这类无信息量的措辞。每个 idea 必须能让人一眼看出它和其他 idea 的方向差异。

## 工作流

### Step 1: Pre-review

调用 `mcp__codex__codex` 检查生成计划：

```
mcp__codex__codex:
  model: gpt-5.5
  prompt: |
    请检查以下 idea brainstorming 计划是否合理：

    研究方向: {用户描述的研究方向}
    约束条件: {算力/时间/偏好等约束}
    执行计划: 生成 6-12 个候选 idea，每个 1-2 段精炼叙述，只给方向性判断

    检查：是否覆盖上游要求？是否自洽？有无遗漏或过度扩展？
```

### Step 2: 生成候选 idea 池

生成 6-12 个候选 idea。每个 idea 1-2 段，覆盖以下内容（用紧凑叙事串联，不分条）：

```
### Idea Ixx: [简短中文标题]

[1-2 段，覆盖：]
- 要解决什么问题？核心洞见是什么？
- 方法方向（一句话）和理论可分析性/数值模拟可验证性（一句话）
- 如果成功，主要贡献是什么？最大风险/不确定性是什么？
```

约束：
- 候选之间必须有**实质方向差异**
- 默认偏向"计算/理论/数值模拟"方向——优先选能形成 theory → simulation 闭环的 idea
- 每个 idea 必须能落到数值模拟验证
- 明显依赖大规模数据采集、重工程实现且缺乏理论增量的方向，降权并注明

### Step 3: Post-review（迭代循环，最多 10 轮）

调用 `mcp__codex__codex` 检查质量：

```
mcp__codex__codex:
  model: gpt-5.5
  prompt: |
    请检查以下 idea pool：

    {idea pool 内容}

    检查：是否覆盖上游要求？是否自洽？有无遗漏或过度扩展？
    额外检查：每个 idea 是否精炼（1-2 段）？是否只给方向性判断而非展开细节？各 idea 方向是否有实质差异？

    若有问题，明确指出并给出修改建议。
```

迭代逻辑：
- 若 review 指出问题 → 修改 → 继续 review（round++）
- 若 review 通过或达到轮数上限 → 结束

**每轮情况汇总**：打印每轮通过/问题数和问题摘要。

### Step 4: 输出确认

输出：
- `00-00-idea-pool.md` 已生成
- 一段话总结最值得推进的 3-5 个方向及理由
- 提示下一步：`/00-01-idea-evaluate`
